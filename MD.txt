All application data has been mapped to a NoSQL structure within the Firebase Realtime Database. The key principle is using unique Firebase-generated IDs for records instead of array indices.

-   **`/users/{user_auth_uid}`**: Stores user profile information. The key is the user's Firebase Authentication UID, creating a secure link between their login and their data. A `role` (`owner` or `member`) is stored here to manage permissions.

-   **Removed `localStorage`:** The `loadData()` and `saveData()` functions have been completely removed.
-   **Real-time Data Sync:** The application now uses real-time listeners (`.on('value', ...)`). When data changes in the Firebase database, these listeners automatically fire, update the local `appState`, and re-render the user interface. This provides the seamless, instant updates required.
-   **Data Modification:** All functions that previously modified the local `appState` array now write directly to the Firebase database using methods like `.update()`, `.push()`, and `.remove()`. -   **Atomic Operations:** For actions that require multiple data changes (like booking a class, which affects both user credits and course bookings), Firebase's multi-path `update()` is used. This ensures that either all changes succeed or none do, preventing data inconsistency.
